### ステージング環境の同時利用を可能にするCI/CD改善

**期間:** 2025年12月

**概要:**
ステージング環境の占有問題を解決するため、複数PRをまとめてデプロイできるGitHub Actionsワークフローを設計・実装した。

**役割:**
課題の発見から設計・実装・チームへの導入まで一貫して担当した。

**使用技術:** GitHub Actions, GitHub CLI

**課題:**

- ステージング環境の利用がSlackでの宣言制で、他のエンジニアの動作確認待ちが発生していた。
- AI活用による開発速度向上に伴い、動作確認フェーズのボトルネックが顕著になっていた。

**内容:**

- PRに `deploy/staging` ラベルを付与し、GitHub ActionsのUIからワークフローを実行する仕組みを構築した。
- GitHub CLIでラベル付きPRリストを自動取得し、deploy/stagingブランチをdevelopから再構築して各PRをマージした。
- まず自チームで運用を開始し実績を得た上で、全チームに展開した。

**工夫した点:**

- コンフリクト発生時は処理を中断し通知して、気軽にラベルを取り外すなどして対応できるようにしたこと。
- 全チームに展開する際は、社内LTで発表し有効性を認識してもらったり、各チーム活用できているかモニタリングし導入支援したこと。

**結果:**

- 複数開発者が並行して動作確認できるようになり、全体的なサイクルタイム短縮に貢献できた。
- まず自チームで運用を開始し、その後全チームに展開した。他チームの多くのエンジニアから、「生産性良くなった、ありがとう」と言ってもらえた。
- チームを超えた動きはマネージャーからも好意的な評価をいただいた。

詳細は下記の記事にまとめています

- [【GitHub Actions】動作確認を同時にできるようにし、サイクルタイムを改善する](https://qiita.com/MandoNarin/items/6dafdb96f1745aaefa6d)

### 大手飲食チェーンの大型契約に関連する開発

**期間:** 2025年4月~2025年6月

**概要:**
大手飲食チェーン（スカイラーク社）との大型契約が決まり、要望されていたデータ連携機能の開発などに従事した。

**役割:**
メンバーとして活動。チーム構成は、正社員のリーダー1名、PdM1名、業務委託2名、私という構成。
アサインされた従業員データ連携領域の開発をほぼ一人で担当した。

**使用技術:** TypeScript(Vue3), Ruby(Rails7, Sorbet), Cloud Storage, SFTP

**課題:**

- 会社全体で取り組む命運のかかった大型案件の開発でプレッシャーの大きい環境であったこと。
- タイトなスケジュールの中、コア機能の開発を問題なくリリースする必要があったこと。

**開発内容:**

- 先方とやり取りしながらCSVのformatを決め、前提となるデータの依存関係などを考慮して連携タイミングを制御した。
- rakeタスクで定期実行の形で専用のPodを立ち上げて実行管理するクラスを実装し、実行結果を記録した。
- 連携処理は、先方のSFTPサーバーに置かれているCSVファイルを読み込み、Cloud Storageに保存した上で自社DBへのbulk import処理を実装した。

**工夫した点:**

- 設計後、合計16個のsub issueに分割しそれぞれの進捗管理をRoadMapで可視化し、日々オンスケかどうか関係者と共有しながら進めた。
- サーバー負荷も監視し、失敗した場合はリトライ処理が走るようにtransactionやupsertにして冪等性を担保した。
- 成功/失敗のメール通知やSlack通知周りを整備した。運用後も失敗時は原因を特定し、手動実行するなどして復旧対応を実施した。

**結果:**

- 当初のスケジュール通り、リリースまで完遂できた。
- 運用後のサポートも実施し、信頼を重ね大型案件の契約延長に貢献できた。
