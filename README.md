### 従業員満足度アンケート機能の開発

**期間:** 2025年12月~2026年2月

**概要:**
従業員満足度調査機能の管理者向け画面を開発した。

**役割:**
デザイン設計から開発をリードし、Bizメンバーとプロジェクトを組み、デザインレビューや要件調整、設計からリリースまでを一貫して担当した。

**使用技術:** TypeScript(Vue3), Ruby(Rails7, Sorbet)

**課題:**

- デザイナーが不足しており、デザイン段階からエンジニア主導で開発をリードすることが要求されていた。

**開発内容:**

- 管理者向けの従業員満足度調査画面をデザイン設計段階からリリースまでリードして取り組んだ。
- デザインを実現するために裏側でどんなAPIやロジックが必要かを同時に検討しながら進めた。

**工夫した点:**

- デザイン設計は、実際にステージング環境で触れるようにまずmock UIを実装し、Biz側の担当者と何度もすり合わせしながら進めたこと。
- デザインはWevoxなどの他社事例を参考にしたり、実際ユーザーは何のデータが欲しいかのヒアリング結果をもとに考案したこと。

**結果:**

- 最終的に自分の提案したものが関係者全員の合意を得てリリースまで完遂できた。

**反省点・学び:**

- 当初は関係者の意見を丁寧に拾おうとするあまり、「あの人はこれが欲しい」「この人はこれが欲しい」と全員の要望を反映したデザイン案を作成してしまった。
- 結果として、責任者（PdM）からデザインのシンプルさや代替機能の観点でフィードバックをいただき、手戻りが発生した。
- この経験から、全員の要望を網羅した案を作る前に、責任者と「何を優先するデザインか（シンプルさか、機能網羅か）」という上位の方針を先にすり合わせるべきだと学んだ。

### ステージング環境の同時利用を可能にするCI/CD改善

**期間:** 2025年11月

**概要:**
ステージング環境の占有問題を解決するため、複数PRをまとめてデプロイできるGitHub Actionsワークフローを設計・実装した。
前提として、各チームに一つ専用のステージング環境と、全体で活用するステージング環境があった。

**役割:**
課題の発見から設計・実装・チームへの導入まで自発的に改善プロジェクトを掲げて担当した。
それまであまり触れてこなかったGitHub Actionsを学べるチャンスだと捉え、すぐに手を挙げた。

**使用技術:** GitHub Actions, GitHub CLI

**課題:**

- ステージング環境の利用がSlackでの宣言制で、他のエンジニアの動作確認待ちが発生していた。
- AI活用による開発速度向上に伴い、動作確認フェーズのボトルネックが顕著になっていた。

**内容:**

- PRに `deploy/staging` ラベルを付与し、GitHub ActionsのUIからワークフローを実行する仕組みを構築した。
- ワークフロー内では、対象PR一覧を取得し、developから作り直した専用ブランチに各PRをmergeしてデプロイする処理を実装した。
- まず自チームで運用を開始し実績を得た上で、全チームに展開した。

**工夫した点:**

- コンフリクト発生時は処理を中断し通知して、気軽にラベルを取り外すなどして対応できるようにしたこと。
- 全チームに展開する際は、社内LTで発表し有効性を認識してもらったり、各チーム活用できているかモニタリングし導入支援したこと。

**結果:**

- 複数開発者が並行して動作確認できるようになり、サイクルタイム（first commitからdevelopへのmergeまで）の平均が20時間以上で不安定だった状態から、17時間程度で安定する水準まで短縮できた。
- まず自チームで運用を開始し、その後全チームに展開した。他チームの多くのエンジニアから、「生産性良くなった、ありがとう」と言ってもらえた。
- チームを超えた動きはマネージャーからも好意的な評価をいただいた。

**反省点・学び:**

- 自主的な横断改善に比重を寄せた結果、目標設定で置いていた通常業務のPR数が一時的に落ち込み、上位マネージャーから生産性への懸念を指摘されたことがあった。
- 直属の上司には状況を共有していたものの、評価を行う上位レイヤーまで仕事の実態を伝えきれていなかったことが原因。横断的に動く際は、着手前に上長と優先順位を明示的にすり合わせること、進捗と影響範囲を上位レイヤーまで含めて可視化することが重要だと学んだ。

### 大手飲食チェーンの大型契約に関連する開発

**期間:** 2025年4月~2025年6月

**概要:**
大手飲食チェーン（業界第二位）との大型契約が決まり、要望されていたデータ連携機能の開発などに従事した。

**役割:**
メンバーとして活動。チーム構成は、正社員のリーダー1名、PdM1名、業務委託2名、私という構成。
アサインされた従業員データ連携領域の開発をほぼ一人で担当した。

**使用技術:** TypeScript(Vue3), Ruby(Rails7, Sorbet), Cloud Storage, SFTP

**課題:**

- 会社全体で取り組む命運のかかった大型案件の開発でプレッシャーの大きい環境であったこと。
- タイトなスケジュールの中、コア機能の開発を問題なくリリースする必要があったこと。

**開発内容:**

- 先方とやり取りしながらCSVのformatを決め、前提となるデータの依存関係などを考慮して連携タイミングを制御した。
- rakeタスクで定期実行の形で専用のPodを立ち上げて実行管理するクラスを実装し、実行結果を記録した。
- 連携処理は、先方のSFTPサーバーに置かれているCSVファイルを読み込み、Cloud Storageに保存した上で自社DBへのbulk import処理を実装した。

**工夫した点:**

- 設計後、合計16個のsub issueに分割しそれぞれの進捗管理をRoadMapで可視化し、日々オンスケかどうか関係者と共有しながら進めた。
- サーバー負荷も監視し、失敗した場合はリトライ処理が走るようにtransactionやupsertにして冪等性を担保した。
- 成功/失敗のメール通知やSlack通知周りを整備した。運用後も失敗時は原因を特定し、手動実行するなどして復旧対応を実施した。

**結果:**

- 当初のスケジュール通り、リリースまで完遂できた。
- 運用後のサポートも実施し、信頼を重ね大型案件の契約延長に貢献できた。

## 副業

**期間:** 2024年12月~現在

**きっかけ:** 友人づての紹介。

**概要:**
大手Gameメディア事業のECサイト開発に従事。新プロダクト立ち上げ期からJoinし、UIリニューアル、クーポン機能の設計開発、決済機能の開発などを担当した。

**使用技術:** Next.js, Ruby on Rails

**クーポン機能の開発内容:**

- DB設計からUIに至るまで、PMやデザイナーの方と連携しながら開発した。
- 初回限定クーポンや購入後に自動発行されるクーポン、先着5名限定クーポンなど、複数種類の発行方式を実現できる仕様を開発した。
- ユニークなクーポンコードを入力して適用する機能を開発した。
- 入力時に利用可否を事前チェックしつつ、決済時にもトランザクション内で再判定と更新を同一処理で行うことで、同時決済時のレースコンディションに対応した。

**決済機能の開発内容:**

- SP.LINK(旧ソニーペイメント)のe-SCOTTを利用して、複数の決済手段に対応するAPIとUIを開発した。（対応決済手段: PayPay, 楽天ペイ, atone, au Payなど）
- 100ページを超える仕様書を読み込み、決済手段ごとに必要なパラメータを整理した上で、暗号化ロジック、決済処理を実装した。
- 決済完了後のcallback処理では、SP.LINKからの決済結果を復号し、実際のアイテム購入処理を実行する流れを実装した。
- アイテム購入処理で失敗した場合にデータ不整合が起きないよう、決済をキャンセルしてロールバックする復旧処理も合わせて実装した。
