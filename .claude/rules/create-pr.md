# PR作成スキル

「PR作成して、タイトルは~~」と指示された場合、以下の手順で `gh pr create` を実行する。

## デフォルト設定

特に指示がなければ下記をデフォルトとする:

| 項目     | 値        |
| -------- | --------- |
| base     | `develop` |
| draft    | `true`    |
| assignee | 自分      |

## タイトルの組み立て

1. 現在のブランチ名からissue番号を抽出する（例: `fix_1234_some_feature` → `1234`）
   - パターン: `[a-zA-Z]+_([0-9]+)_.*`
   - 抽出できない場合はissueなしとして処理する
2. タイトルを `[Project#{ISSUE_NUM}] {ユーザー指定タイトル}` の形式にする
   - issue番号がなければ `[Project] {ユーザー指定タイトル}` の形式にする
   - ユーザー指定タイトルがなければ、直近のコミットから適当にタイトルを組み立ててユーザーに確認を取る

## PR本文テンプレート

`{DESCRIPTION}` には以下を簡潔に記載する:

- 修正内容の説明（変更箇所と何をしたか）
- 関連する既存仕様の説明（変更対象の既存の振る舞い）

`{TESTING_NOTE}` には以下を記載する:

- 動作確認のチェックリストを **3項目くらいまで** 簡潔に記載する
- 1,2項目目は本PRで主に変更した機能の正しい挙動を確認するもの
- 3項目目はリグレッション確認（既存機能が壊れていないか）
- 形式: `- [ ] 確認内容`（Markdownチェックリスト）
- 思いつかない / 自明な変更（型修正等）の場合は省略可。その場合はユーザーに確認を取る
- 内容的に3項目を超えそうな場合も、勝手に増やさずユーザーに確認を取る

```markdown
Deploy Info
---------------------

{PR_TITLE}

{DESCRIPTION}

Issue link or Description
--------------------------
close #{ISSUE_NUM}

Testing note
------------

{TESTING_NOTE}

Look-at point
-------------


Screenshot / GIF
----------------
```

## コミットメッセージの命名規則

```
[#{ISSUE_NUM}] {タイトル} {変更内容の要約}
```

例: `[#1234] ユーザー一覧画面にフィルター追加 APIにパラメータ追加`

- `{タイトル}` はissueまたはPRのタイトルに準ずる
- `{変更内容の要約}` はそのコミットで何をしたかを簡潔に記載
- `Co-Authored-By` 行は付けない

## 手順

1. リモートにブランチが存在するか確認し、なければ `git push -u origin {branch}` を実行する
2. PR作成内容（タイトル・base・branch・draft・assignee・issue num）をユーザーに提示して確認を取る
3. 承認されたら `gh pr create` を実行する

## PRレビュー対応のコメント返信

PR レビューのコメントに対して、対応コミットを参照しつつ返信する場合は以下のルールに従う。

### コミットへの参照リンク形式

返信本文の末尾には、対応コミットへのMarkdownリンクを必ず含める:

```
[コミットメッセージ](https://github.com/{OWNER}/{REPO}/pull/{PR_NUM}/commits/{FULL_COMMIT_SHA})
```

- ハッシュは **フル40文字** を使う（短縮ハッシュではURLのリンクが安定しない）
- `#abc1234` 形式はGitHubではissue/PR番号扱いになるため、コミット参照には使わない
- リンクテキストはコミットメッセージのタイトル行をそのまま使う
- フルSHAは `git rev-parse <short_sha>` または `git log -1 --format=%H` で取得できる

### コメント返信のコマンド

```sh
gh api repos/{OWNER}/{REPO}/pulls/{PR_NUM}/comments/{COMMENT_ID}/replies -X POST -f body="{返信内容}"
```

返信内容には以下を含める:

1. 指摘への対応方針（同意 / 別案 / 却下とその理由）
2. 末尾に対応コミットへのMarkdownリンク
